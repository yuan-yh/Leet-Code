This article analyzes the core process of the operation and maintenance Agent in the architecture design, aiming to automate the repetitive alarm troubleshooting work. Key points include:​

1.

Core goal of operation and maintenance Agent: Transform the alarm handling experience of operation and maintenance personnel into automated processes, and replace manual work in repetitive alarm troubleshooting through a closed - loop of plan generation, tool execution, and dynamic adjustment.​

2.

Plan generation (Planner): Based on alarm types and retrieved operation and maintenance manuals, generate executable multi - step troubleshooting plans in JSON format. Input includes alarm information and handling documents, and output is a structured plan with steps, tool parameters, and expected results.​

3.

Plan execution (Executor): Integrate monitoring and log tools to automatically execute plan steps, such as using query\_prometheus\_alerts to get active alarm details and query\_log to search logs.​

4.

Dynamic re - planning (Replanner): Evaluate the execution results of the Executor. If the result meets expectations, execute the next step; if not, adjust the plan; if the goal is achieved or cannot continue, terminate the plan.​

5.

Example of re - planning decision: When the step 1 result shows no error logs, adjust the keywords and time range; when step 2 indicates a "context cancel" error for less than 30 minutes, terminate the plan.​

6.

Automation capabilities source: The automation of the operation and maintenance Agent comes from the collaboration of structured planning, tool - based execution, and dynamic adjustment.​

7.

Benefits of the architecture: The operation and maintenance Agent can take over 80% of repetitive alarm troubleshooting work, allowing engineers to focus on root - cause analysis and process optimization. ​

计划执行（Executor）：调用工具完成单步排查​

目标 ：通过集成监控/日志工具，自动执行计划中的步骤，替代人工 查日志/查监控 的操作。​

核心工具示例 ：​

1.

告警查询工具（ query\_prometheus\_alerts ）：从Prometheus API获取当前活跃告警详情​

2.

日志查询工具（ query\_log ）：通过腾讯云CLS的MCP服务，用自然语言查询日志（如 搜索ad\_app最近1小时的response error日志 ）​

动态重规划（Replanner）：评估进度并调整策略​

目标 ：根据Executor的执行结果，判断是否需要调整计划（如增加步骤、终止排查），替代人工 持续关注/判断是否需进一步处理 的决策过程。​

评估逻辑 ：​

•成功条件 ：当前步骤结果符合预期（如步骤1返回error日志，步骤2明确原因为 context cancel ），执行下一步​

•调整条件 ：结果不符合预期（如日志查询无结果），触发计划修正（如扩大时间范围、更换关键词）​

•终止条件 ：达到目标（如确认 下游发版导致临时超时，无需处理 ）或无法继续（如需人工介入下游沟通）​

总结​

运维Agent的自动化能力源于三方面协同：​​

•结构化规划 ：Planner将模糊的运维经验转化为可执行步骤，降低复杂告警的处理门槛•​

•工具化执行 ：Executor集成监控/日志系统，替代人工重复操作，提升响应速度​

•动态的调整 ：Replanner根据实时结果修正计划，适配 下游发版/临时抖动 等不确定性场景​

​

通过这套架构，运维Agent可接管80%的重复性告警排查工作，让工程师聚焦于根因分析和流程优化。​
